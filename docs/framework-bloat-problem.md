# 框架肥大問題：現況、解法、目標

> 本文件是研究與討論的載體，不是框架機制。刻意保持樸素（無 frontmatter、無 ticket、無 tracking），放在專案層而非 `.claude/` 框架層。內容是「我的視角下的提案」，不是定論——解法是假說，需要一起驗證。可隨時刪除或重寫。
>
> 產出脈絡：app_tunnel（框架孵化器）使用框架幾小時後，使用者觀察到「框架持續肥大、無法解決」，要求從 AI 視角找方法。本文件記錄該次診斷。

---

## 一句話問題陳述

框架只有「累積機制」、沒有「代謝機制」，因此必然單向膨脹——大到使用者無法掌握、連 AI 都無法可靠遵守，最終反過來損害它原本要保障的可靠性。

---

## 現況（證據，非推測）

- **活標本**：在 app_tunnel 使用框架的這幾小時，寫了 **0 行產品 code**，卻淨增約 20 個 artifact（11 需求文件 + 3 ticket + 4 memory + 1 缺口記錄 + 一串待辦）。
- **每個「發現」都生一個追蹤物**：發現問題 → 建 ticket / 記 memory / 寫缺口，從不刪減。
- **自我複製**：連「討論如何不肥大」這件事本身，都在排隊生產新 artifact。
- **既有警告被忽略**：session 啟動時 hook already 報「auto-load 集合 45.5k / 45k tokens 超標」「file-size 超標 36 個」——框架已經知道自己超重，但那只是警告，被無視。
- **AI 已遵守不能**：這幾小時 AI 違反了多條它「本該知道」的規則、confabulate 了工具結果、寫壞工具呼叫——因為規則多到無法全部遵守。為「讓 AI 可靠」而建的框架，大到讓 AI 無法可靠遵守它。

---

## 根因（為什麼是結構性必然，不是自律問題）

1. **只有累積器，沒有代謝器。** quality-baseline 規則 5：「發現任何問題，無論優先級，都必須建立 Ticket，發現即建立，不詢問」。continuous-learning 捕獲 memory、error-pattern 記錄、缺口回饋——每條都把「發現」導向「新增」。沒有任何一條規則導向「刪除 / 合併 / 淘汰」。只有輸入沒有輸出的系統，熵單向增加。

2. **「發現即記錄」把「注意到」等於「值得永久保存」。** 規則 5 的「不詢問確認」刪掉了「這值不值得追蹤」的判斷，把每個瞬時念頭都寫進長期記憶。但健康的記憶系統靠「遺忘」運作——遺忘是特徵，不是 bug。

3. **加易刪難的不對稱。** 加防線：立即收益（安心）+ 理由現成。刪防線：立即風險（萬一有用）+ 要舉證「它沒用了」。理性的人每次都選加、不敢刪。同構於法典膨脹、監管累積、技術債。

4. **自我拆台。** 框架越大，AI 與使用者越無法完整掌握；超過某點，框架複雜度本身成為新的錯誤來源。

5. **更深的質疑：「框架該持續成長」這個假設本身可能就是錯的。** 成熟方法論的標誌不是越來越大，而是收斂到越來越少、越來越高槓桿的原則。持續變大，更可能是「還沒找到核心原則」的症狀，而非「越來越完備」。Saint-Exupéry：「完美不是無可增加，而是無可刪減。」健康的框架演化，後期應是淨收縮。

---

## 解法（核心：反轉預設）

**核心原則**：把預設從「保留」反轉成「過期」。
從「預設永久，證明無用才能刪」→ 改成「**預設過期，證明有用才能留**」。
這讓「留下」也需要舉證，與「加上」對稱，直接攻擊根因 3 的不對稱。

**操作（全部是改造現有機制使其自我代謝，不新增任何東西）**：

1. **預算警告 → 硬約束（forcing function）。** 框架已有預算機制（auto-load token、file-size），目前只是警告。升級為硬約束：**超預算就不能加新的，除非先刪掉等量的舊的**。晉升變成零和。

2. **對話產物預設蒸發。** 對話是探索，大部分產物應該消失。發現 / 待辦 / 想法預設只活在對話裡（隨 context 蒸發），除非通過高門檻落地閘門——三問：「三個月後還有人 care 嗎？是否已被別的涵蓋？刪掉它會有人受傷嗎？」三問過不了，不落地。這是規則 5 的對偶。

3. **追蹤物自證存活。** ticket / memory / 缺口記錄過期後自動進「待淘汰」，要主動證明還活著才保留。反轉舉證責任。

**對規則 5 的修正**：規則 5「發現即追蹤」是肥大的主要引擎，必須配一條對偶——「被追蹤的東西要定期自證存活，否則自動淘汰」。

---

## 試金石（檢驗任何解法）

**任何「處理肥大」的方案，如果它本身是「新增一個機制 / 規則 / skill / hook」，它就已經失敗了——以肥治肥。**

真正的解法只能是：減法、反轉預設、或改造現有機制。上面三個操作全部是「改造 / 強化現有」，沒有一個是「新增」。若日後忍不住想為「反肥大」建一個新系統，那正證明這個病已深入骨髓。

---

## 目標

- **短期**：讓框架能「代謝」——裝上遺忘 / 淘汰 / 過期機制（透過改造現有，不新增）。
- **中期**：框架總量「淨收縮」而非淨增長；每次晉升一個，先淘汰一個。
- **終極**：收斂到少數高槓桿原則——以「無可刪減」為成熟標誌，而非「無可增加」。
- **驗證場**：app_tunnel（孵化器）先驗證瘦身機制是否可行、是否傷到真防線，確認後再 sync-push 到 claude.git 散佈。

---

## 待解問題（留給後續討論，刻意不轉成待辦清單）

- 「預設過期」的期限與觸發條件，怎麼定才不會變成又一套複雜流程（自我打臉）？
- 硬約束會不會擋掉「真正該加」的東西？如何在不新增判斷機制的前提下區分真防線與冗餘？
- 規則 5 是核心引擎，改它影響所有用框架的專案，風險高——怎麼安全地改、先在 app_tunnel 驗證什麼？
- 如何分辨「框架還沒成熟所以該長」與「框架已過度成長所以該縮」？有沒有客觀信號？
