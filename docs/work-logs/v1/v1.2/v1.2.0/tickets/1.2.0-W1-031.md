## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`version-release check`（不帶 `--version`）在本專案（monorepo: Flutter app/ + Go server/）失敗，兩個直接原因：

1. **版本偵測誤選 pubspec build 版**：`.claude/.version-release.yaml` 指定 `version_source.primary: app/pubspec.yaml`，其值為 `1.1.0+2`（含 Flutter build number 後綴）；但實際開發中版本為 `1.2.0`（`docs/work-logs/v1/v1.2/v1.2.0/` 已建立）。`detect_version()` 直接採版本源 `1.1.0+2`，無視更高的 in-dev worklog 版本。
2. **preflight 對 +build 後綴崩潰**：`check_previous_versions_completed("1.1.0+2")` 在 `tuple(int(p) for p in version.split("."))` 對 `"0+2"` 執行 `int()` → `ValueError: invalid literal for int() with base 10: '0+2'`，preflight 直接 traceback 中斷。

### 影響範圍

- `.claude/skills/version-release/scripts/version_release.py`：`normalize_version`、`detect_version`、`compare_semantic_versions`、`check_previous_versions_completed`。
- `.claude/.version-release.yaml`：補註版本來源語意。
- 任何使用 Flutter pubspec `X.Y.Z+build` 版本格式的 monorepo 專案皆受影響。

### 相關 Error Pattern

無直接對應 error-pattern。屬輸入格式（SemVer build metadata）未在版本拆分解析點防禦的實作缺口。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

三項修正（皆在 `version_release.py`）：

1. **+build 後綴剝離**：新增 `strip_build_metadata(version)`，切除 `+build` 與 `-prerelease` 後綴取核心 `X.Y.Z`。套用於三處拆分解析點：
   - `normalize_version`（顯式 `--version 1.1.0+2` 亦防禦）
   - `compare_semantic_versions`（v1/v2 比較前剝離）
   - `check_previous_versions_completed`（原崩潰點，`int(...)` 前剝離）
2. **monorepo in-dev 偵測**：新增 `detect_indev_worklog_version(root, pattern)`，依 `worklog_path_pattern` 遞迴掃描 worklog 目錄取最高 `vX.Y.Z`。`detect_version()` 比較版本源版本與 in-dev 版本，採較高者；版本源落後時輸出 INFO 並改採 worklog 版本。
3. **monorepo 版本模型**：`.claude/.version-release.yaml` 補註——app(flutter) 為版本 SoT（pubspec 含 +build，偵測自動剝離）、server(go) 為 tag-driven、in-dev worklog 覆寫語意。未新增 monorepo subprojects schema（既有單一產品版本號模型已足夠，避免過度設計）。

無需 spawn 其他 ticket。
