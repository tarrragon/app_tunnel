## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP

- W：擴增出三種缺口類型 (a)規則不存在 (b)存在但無強制 wiring (c)存在但 opt-in 無觸發。Consider the Opposite：「假設框架其實有強制呢？」→ 實證證實有（style-guardian 註冊在案），翻轉了「缺規則」直覺。
- R：見「重現實驗結果」。基本率——edit-time linter 對「缺席」普遍失明（業界 linter 多檢查既有 code 的 pattern，非「應有而無」的架構缺席）。
- A：補強制 wiring 一次性投入，回報是未來每個新專案/功能不重蹈 v1.0。
- P：行前預想——若只補 hook 不補 phase gate，下個 greenfield 專案仍可能在 hook 觸發前就累積大量硬編碼（hook 是 edit-time 點防禦，覆蓋不了「整個功能週期沒人 bootstrap 設施」）。故方案須含 phase/acceptance 層。

### 根因（一句話）

框架的品質閘門驗證「既有設施的一致性」，但沒有閘門偵測「應有設施的缺席」；且強制只綁 edit-event、不綁 phase/acceptance。greenfield 專案從不 bootstrap i18n/theme，於是依賴那些設施存在才生效的 hook 終生空轉。

### 與 009 的一體兩面

009 發現框架對「出過事的（PC-XXX）」過度設防（259 inline gate 增生）；015 發現對「沒出過事的常規品質」設防失效（absence-blind + warning-only）。同一根因：**框架的強制是 incident-driven，不是 standard-driven**。出事才長閘門；常規品質規範因從未釀成被追蹤的 incident，永遠等不到屬於它的閘門。

## 重現實驗結果
### 框架現況 grep 實證

**既有強制設施（存在且註冊）**：
- `style-guardian-hook.py`：PreToolUse on Edit/Write，偵測 `.dart`，含 `SizedBox(height/width: \d+)` → Spacing 違規規則，但輸出為 **WARNING**（"X potential violations detected"，exit 0 不阻擋）。且預設專案有 `ui_config.dart`/`theme.dart`/`UISpacing` 設施可對照。
- `l10n-sync-verification-hook.py`：**僅在編輯 `.arb` 檔時觸發**，目的是驗證 ARB↔gen-l10n 生成檔同步。
- `output-style-check.py`：檢查回應格式 output style，與程式碼品質無關。

**缺口證據**：
- TDD flow / verification-framework / acceptance-criteria methodology / ticket-body-schema → **零提及** i18n/硬編碼/style（grep 全空）。
- app 實況：`core/constants` 空、無 `theme.dart`、無任何 `.arb`。

### 已驗證事實 vs 假設

- 假設「框架缺規則」→ **否證**：style-guardian + l10n-sync 都存在且註冊。
- 真根因（已驗證）：兩道 hook 都是「**存在物的一致性檢查**」，對「**應存在物的缺席**」結構性失明：
  - l10n-sync 只在 `.arb` 編輯時觸發 → 專案從未建 i18n、永遠不碰 .arb → hook 終生不觸發。硬編碼 21 處英文字串可暢行無阻。
  - style-guardian 雖 on .dart 觸發，但 (a) WARNING-only 可忽略，(b) 預設 UISpacing/theme 設施存在才有對照基準，本專案從未 bootstrap → 只能空轉嘮叨。
- 第二軸：所有強制都綁「檔案編輯事件（PreToolUse）」，無任何「phase gate / acceptance 條目」在功能驗收時強制問「本功能引入的文字/顏色/數字是否已集中」。
