評估 框架品質規範缺口（i18n/常數集中/UI美感為何 v1.0 全程未攔截）

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP

- W：擴增出三種缺口類型 (a)規則不存在 (b)存在但無強制 wiring (c)存在但 opt-in 無觸發。Consider the Opposite：「假設框架其實有強制呢？」→ 實證證實有（style-guardian 註冊在案），翻轉了「缺規則」直覺。
- R：見「重現實驗結果」。基本率——edit-time linter 對「缺席」普遍失明（業界 linter 多檢查既有 code 的 pattern，非「應有而無」的架構缺席）。
- A：補強制 wiring 一次性投入，回報是未來每個新專案/功能不重蹈 v1.0。
- P：行前預想——若只補 hook 不補 phase gate，下個 greenfield 專案仍可能在 hook 觸發前就累積大量硬編碼（hook 是 edit-time 點防禦，覆蓋不了「整個功能週期沒人 bootstrap 設施」）。故方案須含 phase/acceptance 層。

### 根因（一句話）

框架的品質閘門驗證「既有設施的一致性」，但沒有閘門偵測「應有設施的缺席」；且強制只綁 edit-event、不綁 phase/acceptance。greenfield 專案從不 bootstrap i18n/theme，於是依賴那些設施存在才生效的 hook 終生空轉。

### 與 009 的一體兩面

009 發現框架對「出過事的（PC-XXX）」過度設防（259 inline gate 增生）；015 發現對「沒出過事的常規品質」設防失效（absence-blind + warning-only）。同一根因：**框架的強制是 incident-driven，不是 standard-driven**。出事才長閘門；常規品質規範因從未釀成被追蹤的 incident，永遠等不到屬於它的閘門。

## 重現實驗結果
### 框架現況 grep 實證

**既有強制設施（存在且註冊）**：
- `style-guardian-hook.py`：PreToolUse on Edit/Write，偵測 `.dart`，含 `SizedBox(height/width: \d+)` → Spacing 違規規則，但輸出為 **WARNING**（"X potential violations detected"，exit 0 不阻擋）。且預設專案有 `ui_config.dart`/`theme.dart`/`UISpacing` 設施可對照。
- `l10n-sync-verification-hook.py`：**僅在編輯 `.arb` 檔時觸發**，目的是驗證 ARB↔gen-l10n 生成檔同步。
- `output-style-check.py`：檢查回應格式 output style，與程式碼品質無關。

**缺口證據**：
- TDD flow / verification-framework / acceptance-criteria methodology / ticket-body-schema → **零提及** i18n/硬編碼/style（grep 全空）。
- app 實況：`core/constants` 空、無 `theme.dart`、無任何 `.arb`。

### 已驗證事實 vs 假設

- 假設「框架缺規則」→ **否證**：style-guardian + l10n-sync 都存在且註冊。
- 真根因（已驗證）：兩道 hook 都是「**存在物的一致性檢查**」，對「**應存在物的缺席**」結構性失明：
  - l10n-sync 只在 `.arb` 編輯時觸發 → 專案從未建 i18n、永遠不碰 .arb → hook 終生不觸發。硬編碼 21 處英文字串可暢行無阻。
  - style-guardian 雖 on .dart 觸發，但 (a) WARNING-only 可忽略，(b) 預設 UISpacing/theme 設施存在才有對照基準，本專案從未 bootstrap → 只能空轉嘮叨。
- 第二軸：所有強制都綁「檔案編輯事件（PreToolUse）」，無任何「phase gate / acceptance 條目」在功能驗收時強制問「本功能引入的文字/顏色/數字是否已集中」。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 缺口類型判定（a 不存在 / b 存在無 wiring / c opt-in 無觸發）

| 品質規範 | 既有設施 | 類型 | 精確診斷 |
|---------|---------|------|---------|
| i18n（文字集中） | l10n-sync-verification-hook | **b 變體：presence-blind** | hook 只在 .arb 編輯時觸發，偵測 sync 不偵測缺席；無 i18n 專案永不觸發 |
| 數字/常數集中 | style-guardian-hook（含 SizedBox 規則） | **b：warning-only + infra-依賴** | 觸發但不阻擋，且預設 UISpacing/theme 存在；本專案未 bootstrap → 空轉 |
| UI 美感 | impeccable skill + style-guardian | **c：opt-in 無觸發** | impeccable 是手動 skill 無任何 phase 強制；style-guardian 只查機械 pattern 非美感 |

### 最小強制 wiring 方案（三選一可組合，建議全採）

1. **Acceptance 層（最高槓桿、最低成本）**：在 `acceptance-criteria-methodology` + `ticket-body-schema` 為「含 UI 的 IMP」加一條標準驗收維度——「本功能引入的 user-facing 文字/顏色/魔術數字已集中（i18n key / theme token / 常數）」。讓常規品質成為 **per-feature phase gate**，不再只靠 edit-time。
2. **Presence-detection hook**：新增/擴充 hook 在 .dart Write/Edit 偵測「硬編碼 user-facing 字串 + 裸 Color() + 魔術數字」且**升級為 blocking-with-override**（非純 warning）；針對 greenfield「缺席」而非只查既有 infra 一致性。
3. **TDD flow 接線**：Phase 1（功能設計）或 Phase 4（重構評估）明列 i18n/常數/設計 token 檢查點。

### Spawn IMP 規劃（PM 落地）

| # | 標題 | 類型 | 備註 |
|---|------|------|------|
| C1 | acceptance 模板 + ticket-schema 加「集中化」驗收維度 | DOC/IMP | 方案 1，最高槓桿，先做 |
| C2 | presence-detection hook（硬編碼/裸 Color/魔術數字，blocking-with-override） | IMP | 方案 2，需 basil-hook-architect |

### 跨 ANA 綜合（013/014/015 統一排序）

關鍵 insight：013-IMP1（bootstrap i18n/ARB）+ 014-IMP1（bootstrap theme token）**正是讓那兩道空轉 hook 開始生效的前提**——一旦 .arb 與 theme.dart 存在，l10n-sync 與 style-guardian 就有對照基準。故統一排序建議：
1. **先 prevention（C1 acceptance 維度）**——立刻生效，防新功能再漏。
2. **再 bootstrap（013-IMP1 i18n 設施 + 014-IMP1 token 常數檔）**——這同時「點亮」既有 hook。
3. **再 remediation（013 文字/數字/顏色遷移、014 各畫面重設計）**——此時新 acceptance 維度 + 點亮的 hook 會驗證遷移成果、防回退。
4. C2（presence hook）可與 remediation 並行。
