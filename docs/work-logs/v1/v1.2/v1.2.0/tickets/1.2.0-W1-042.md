## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 分析結論

依 domain-bundle-mapping 方法論從 5 個 domain spec 的 FR 反推 bundle 邊界：

**跨語言邊界 DAG**：
```
connectivity (deploy) ──→ auth (外部設定) ──→ proxy (Go server)
                                                     │ WS
                                              client (Flutter) ←── enrollment (Go, 一次性)
```

**真 domain bundle（含自有計算邏輯）**：

| Domain | Bundle | 分類 | 實現端 |
|--------|--------|------|--------|
| client | TerminalProtocol | supporting VO | Flutter app |
| enrollment | CredentialBundle | aggregate root | Go server |
| proxy | ReverseProxy | aggregate root | Go server |

**無自有 domain 計算的 domain（策略宣告/部署設定）**：
- auth：認證行為完全委託 Tailscale + ttyd，所有 FR 歸 infra
- connectivity：連線行為完全委託 Tailscale daemon，所有 FR 歸 infra/deploy

**關鍵發現**：
1. 無 shared kernel：三個 domain 的計算互不消費，跨域共用的是契約格式（contract.md）
2. 跨語言邊界清晰：Flutter app 只有 client；Go server 有 proxy + enrollment
3. 契約介面三組：WS 子協議（client↔proxy）、憑證包 JSON（enrollment→client）、HTTP auth header（client→proxy→ttyd）

### 產出物

5 個 domain-map.md 已建立：
- `docs/spec/auth/domain-map.md`
