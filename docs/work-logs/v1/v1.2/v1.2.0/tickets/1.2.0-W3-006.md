## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

unit test 用 FakeWebSocketChannel，遮蔽了 text/binary frame 差異和 auth token 真實行為。需要對真實 ttyd WS 連線驗證協議正確性。

### Context Bundle

ttyd 已安裝：/opt/homebrew/bin/ttyd v1.7.7。
測試策略：test fixture 在 setUp 啟動 ttyd（隨機 port，帶 basic auth），tearDown 殺掉。Dart test 透過 IOWebSocketChannel 連真實 WS。
驗證項：(1) WS 握手成功（101 Switching Protocols）(2) auth token frame 正確發送後收到 shell prompt (3) text frame 使用（非 binary）。
注意：ttyd 子協議為 tty，握手需 protocols: ['tty']。auth token 格式為 base64(user:pass) 的 JSON frame。
現有 integration test 位置：app/test/integration/（已有 connection_flow_test.dart 等）。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

建立 `app/test/integration/protocol_integration_test.dart`，包含：

1. **TtydFixture**：管理真實 ttyd process 生命週期（隨機 port、`--writable`、`--credential`、setUp 啟動 / tearDown 殺掉）
2. **4 個測試**：
   - WS 握手成功（tty subprotocol + basic auth）
   - WS 握手拒絕錯誤憑證
   - Auth token frame 認證並啟動 shell
   - Input binary frame round-trip（echo 指令 → 收到 output）

**協議發現**（integration test 揭露的真實行為）：
- ttyd v1.7.7 server→client 是 **binary frame**（`List<int>`），非 text frame — unit test FakeWebSocketChannel 假設 String 型別遮蔽了此差異
- client→server input 必須是 **binary frame**（`List<int>`），text frame 被忽略
- ttyd 預設 **readonly**，需 `--writable` 旗標否則靜默丟棄 input
- auth token 值為 `base64(user:pass)` 格式
