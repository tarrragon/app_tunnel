## 重現實驗結果
### WRAP R 階段：讀雙腳本後的根因（世界平面）

- **sync-pull 已有撞號處理**：`sync-claude-pull.py` 行 1329-1341「PC 編號撞號偵測與自動重編號」+ `_next_available_pc_number`。PC-181/182 即此產物（pull 上游 PC-177-defensive 撞本地 → 自動重編 181）。
- **sync-push 無對稱邏輯**：full-overlay 推全部，含本地原生 177/178（malformed/ticket-body）+ pull 重編 artifact 181/182 → 上游 6 檔 mess（GitHub API 直查確認：PC-177×2、178×2、181、182）。
- **lineage 標記機器可讀**：PC-181 檔頭「上游編號 PC-177，本專案重編 181」——push 本可據此辨識 artifact 不外推。
- 命名空間慣例存在但少用（PCAPP-001/002、PCTUNL-001 共 3 條）。

### 已驗證根因（一句）

不是「缺碰撞偵測」——pull 端有。是 **sync-push 與 pull 不對稱**：pull 進來去撞號重編、push 出去無腦 overlay。PC-APP-002 早記「於 sync-push 時由框架賦予 canonical 編號」指了此方向，未實作。
