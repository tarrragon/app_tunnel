## 重現實驗結果
### WRAP R 階段：讀雙腳本後的根因（世界平面）

- **sync-pull 已有撞號處理**：`sync-claude-pull.py` 行 1329-1341「PC 編號撞號偵測與自動重編號」+ `_next_available_pc_number`。PC-181/182 即此產物（pull 上游 PC-177-defensive 撞本地 → 自動重編 181）。
- **sync-push 無對稱邏輯**：full-overlay 推全部，含本地原生 177/178（malformed/ticket-body）+ pull 重編 artifact 181/182 → 上游 6 檔 mess（GitHub API 直查確認：PC-177×2、178×2、181、182）。
- **lineage 標記機器可讀**：PC-181 檔頭「上游編號 PC-177，本專案重編 181」——push 本可據此辨識 artifact 不外推。
- 命名空間慣例存在但少用（PCAPP-001/002、PCTUNL-001 共 3 條）。

### 已驗證根因（一句）

不是「缺碰撞偵測」——pull 端有。是 **sync-push 與 pull 不對稱**：pull 進來去撞號重編、push 出去無腦 overlay。PC-APP-002 早記「於 sync-push 時由框架賦予 canonical 編號」指了此方向，未實作。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### sync-push 撞號對稱設計（鏡像既有 pull 邏輯，非新建）

讓 sync-push 對稱於 sync-pull 的撞號處理，三條規則：

1. **辨識 pull 重編 artifact 不外推**：push 前掃本地 error-pattern，凡檔頭含 lineage 標記「上游編號 PC-NNN，本專案重編 PC-MMM」者 = pull 端撞號產物（canonical 版已在上游 PC-NNN），**排除不推**（防 181/182 類重複）。
2. **本地原生撞號賦 canonical 號**：本地原生 pattern（無 lineage、bare PC-NNN）若其號已被上游不同 slug 佔用 → push 時賦 next-available canonical 號（鏡像 pull 的 `_next_available_pc_number`），並回寫本地 + lineage 標記。
3. **首跑對帳清 mess**：此修第一次執行時，依規則 1/2 自動辨識當前上游 6 檔 mess——還原 canonical 177-defensive/178-ui-test、移除誤推的 181/182 + 把 malformed/ticket-body 賦新 canonical 號。取代手動刪（避免 PC-APP-002 過刪 + 我再手動出錯）。

設計積木全現成：pull 的 `_next_available_pc_number` + lineage 標記格式 + GitHub API 查上游佔用。非從零。
