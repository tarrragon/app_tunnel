## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

VIBE CODING 預設美術風格產生「無設計決策」的介面：5 個 screen 全部依賴 Flutter 預設 Material 3 元件（`AppBar` + `Center` + `Column` + 預設按鈕），無設計 token 系統、無一致的色彩/間距/字體階層決策。終端機相關畫面則是各自硬編碼深色值（`Color(0xFF1E1E1E)`、`Colors.white70` 等），與 Material screen 的淺色主題分裂為兩套互不相干的視覺語言。

根因可拆三層：
1. **無設計 token 層**：`main.dart` 只有 `colorSchemeSeed: Colors.blue + useMaterial3`，其餘畫面顏色/間距/字級全是行內硬編碼或 Material 預設，無單一來源。
2. **兩套割裂視覺語言**：home/enrollment/qr_scanner 用 Material 預設淺色主題；terminal screen + renderer 用硬編碼深色（`0xFF1E1E1E` 背景 / `white70` 文字）。使用者從首頁進終端會經歷主題斷層。
3. **ANSI 配色未經設計**：`ansi_parser.dart` 的 16 色盤是 xterm 經典值（red=`0xFFCD0000` 等），未針對本 app 的深色背景做對比度校準，也未與 app 品牌色協調。

### 影響範圍

| 檔案 | 角色 | 現況硬編碼 |
| ---- | ---- | ---- |
| `app/lib/main.dart` | 全域主題 | `colorSchemeSeed: Colors.blue`、`useMaterial3: true`（無自訂 ThemeData） |
| `app/lib/shared/widgets/home_screen.dart` | 首頁 | 全 Material 預設、`SizedBox(height: 24)` |
| `app/lib/features/enrollment/screens/enrollment_screen.dart` | 配對 | `Colors.green`/`Colors.red` 狀態色、`Icon size: 96`、間距 16/24 |
| `app/lib/features/enrollment/qr_scanner_screen.dart` | QR 掃描 | `Colors.black.withAlpha(128)` 遮罩、掃描框 `250x250`、`circular(16)` |
| `app/lib/features/terminal/screens/terminal_screen.dart` | 終端主畫面 | `Color(0xFF1E1E1E)` 背景、`Colors.white70`/`white54`/`redAccent`、fontSize 16、間距 16/24/48 |
| `app/lib/features/terminal/renderer/terminal_renderer.dart` | 終端渲染 | 前景 `0xFFE0E0E0`、背景 `0xFF1E1E1E`、fontSize 14、padding 8 |

## 重現實驗結果

### 實驗方法

本 ANA 的「問題」是美感缺陷，重現方式為實際讀取 7 個畫面碼，以 impeccable skill 的 product register 評估框架逐項對照其規則，記錄違規事實。非執行 app（單人自用工具、無 UI 自動截圖環境），改以「讀碼 + 規則對照」作為等價重現——每條斷言均指向具體檔案行號可複查。

工具：impeccable skill（已 invoke，載入 product.md register + 共同設計規則 + palette seed），WRAP 完整模式。

### 實驗執行

實際讀取的檔案（全部已讀、非推測）：

1. `main.dart`：`colorSchemeSeed: Colors.blue` + `useMaterial3: true`，無 `darkTheme`，無自訂 ThemeData。
2. `home_screen.dart`：`AppBar(title: 'App Tunnel')` + 置中 Column（`Text('App Tunnel - Remote Terminal')` + `SizedBox(24)` + `FilledButton.icon`）。
3. `enrollment_screen.dart`：96px qr icon + 說明文字 + `ElevatedButton.icon`；狀態訊息 `Colors.green`/`Colors.red`（L100ms）。

### 實驗發現

**已驗證事實**（讀碼直接觀察）：

1. 全 app 零設計 token 層——顏色、間距、字級 100% 行內硬編碼或 Material 預設。
2. 視覺系統二分裂：4 淺色 Material 畫面 vs 1 深色終端，無共用語彙、無過渡。
3. 主操作按鈕三種 widget（FilledButton/ElevatedButton x2），component vocabulary 違規。
4. 狀態色（綠/紅）對比未驗證，淺底綠字高機率不達 4.5:1。
5. ANSI 16 色為 xterm 經典硬編碼，未對 `#1E1E1E` 背景校準、未與 app 品牌協調。
6. 終端 3 個非連線狀態畫面結構重複，可抽象為單一 status view。

**仍未驗證的假設**（需 IMP 階段或實測確認，不阻擋方向定案）：

- 各 ANSI 色在新背景的實際對比值需 IMP 階段以對比公式逐色驗證（本 ANA 給目標值，不取代逐色實測）。
- 終端使用者對「重配 ANSI 色」的接受度（擬真度偏好）——取捨章已用「語意保真 + 僅校準明度/彩度」降低風險，但個人偏好屬未驗證，故設計為 token 可回退原始 xterm 值。

### W — 設計方向候選（擴增選項）

| 方案 | 描述 | 取捨 |
|------|------|------|
| A. 維持現狀（Material 預設藍） | 不重設計，只把硬編碼集中 | 零美感提升；違背 ticket why |
| B. 淺色 Material + 自訂 seed | 全 app 統一淺色主題，終端仍深色 | 仍二分裂；終端與 app 主題斷層不解 |
| **C.（採用）深色為主、cobalt 品牌 + amber accent 的「儀表艙」單一語言** | 全 app 深色 theme，終端是這套語言的原生延伸而非異類；ANSI 在同一深背景下語意保真校準 | 解決二分裂與品牌缺席；終端與 app 終於同源 |
| D. 雙主題（亮/暗可切） | 同時做亮暗兩套 | 單人自用工具過度工程；先做暗色（終端工具本質） |

**選定 C 的理由（scene sentence）**：「凌晨值機艙——深 cobalt 夜空下的儀表輝光，精確而安靜地專注」。本 app 是隨身終端工具，多在低光環境單手操作，深色為原生選擇（非「工具看起來酷所以暗」，而是使用場景強制）。終端本就深色，全 app 深色才能讓終端不再是異類。品牌溫度由 cobalt primary + amber accent + 字體承載，背景保持近黑微冷染（非暖奶油 AI 預設）。

Category-reflex 二階檢查：終端工具的一階反射是「黑底綠字 matrix」；二階反射是「terminal-native dark 配霓虹綠」。本方案避開兩者——背景近黑微冷（非純黑）、品牌色是 cobalt（非綠）、accent 是 avionics amber（信號對比，非裝飾霓虹），ANSI 綠保留但僅作 SGR 語意非品牌色。

### R — 實境檢驗（已於重現實驗完成）

基本率：成熟終端 App（Termius / Blink / iTerm2 主題）多採「深色中性背景 + 單一品牌強調色 + 校準過的 ANSI 16 色（非生 xterm 值）」。本方案與此基本率一致。鋼鐵人測試（被放棄的 B 之優點）：淺色對日間戶外可讀性佳——已以「token 可擴充亮色主題（方案 D 延後）」承接，不否定。

### A — 機會成本

執行重設計 IMP 會擠壓 server 端功能。但本 ANA 僅產出 token + 方向，不阻塞；IMP 落地為獨立 ticket，PM 可依優先級排於 server 之後。10/10/10：10 個月後，有設計 token 層讓後續每個新畫面成本下降，回報持續。

### P — 準備好犯錯（行前預想）

| 失敗預想 | 防護 |
|---------|------|
| ANSI 重配破壞終端可讀性/語意 | 取捨章定「語意保真」鐵則 + token 保留原始 xterm 值為 fallback 常數 |
| token 與 ANA-A 常數檔結構對不上 | 本表已給「目標 Dart 常數名 + 值」，ANA-A 直接 1:1 承接 |
| 對比未達標 | 所有 body/狀態色標注目標對比，IMP 階段逐色驗證為 acceptance |

### 關鍵取捨：終端輸出區「擬真度 vs 美感」

**問題**：重配 ANSI 16 色是否破壞終端可讀性與 ANSI 語意？

**鐵則（保真不可破）**：

1. **色相語意不動**：SGR 31 永遠映射「紅族」、32 映射「綠族」…16 個索引位置與色相歸屬完全保留。重配只動 L（明度）與 C（彩度），不動 H（色相，±各色仍落在原色相帶內）。程式 `standardColors[code-30]` / `brightColors[code-90]` 的索引邏輯零改動，只換常數值。
2. **normal vs bright 明度序保留**：bright 系列 L 一律高於對應 normal（如 br-red L0.76 > red L0.65），保留終端「bold/bright 更亮」的視覺慣例，不破壞程式對 SGR 1（bold）與 90-97（bright）的語意區分。
3. **背景前景對比達門檻**：所有前景色對 `kColorBg`（`0xFF14171C`）≥4.5:1（normal）/ ≥7:1（bright），優於原 xterm（如 blue `0xFF0000EE` on 近黑僅約 2:1，原本就難讀）——**重配實際提升可讀性**。
4. **可回退**：token 同時保留原始 xterm 值為 `kAnsiXtermFallback*` 常數（不啟用，僅備存），若使用者反映「不像真終端」可一鍵切回。降低 P 階段「個人偏好」未驗證風險。

**為何不破壞美感與語意可共存**：原 xterm 值是 1990 年代 CRT 校準的遺產，在現代深色 LCD 上偏刺眼且部分（blue）對比不足。重配在「保留色相身分」前提下統一明度/彩度到一致的感知亮度帶，使 16 色既各自可辨（語意保真）、又同屬一個調和的儀表盤色系（美感）。語意載體是「色相 + 明度序」，重配兩者皆守，故語意零損；美感載體是「彩度一致性 + 背景對比」，重配兩者皆改善。
