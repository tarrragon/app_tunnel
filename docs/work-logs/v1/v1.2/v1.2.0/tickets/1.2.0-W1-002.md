### 設計：unified-monorepo 模式

`project_type: monorepo` + 頂層 `version_source`（無 `subprojects`）= **統一版本模式**：version_source 指定的子專案版本檔為產品 SoT，tag `v{version}` 標整個 repo、bump 該檔。與既有 `subprojects`（獨立版本）並存、**向後相容**（既有 monorepo 配置用 subprojects，不受影響）。

### 五項改動（檔案：`skills/version-release/scripts/version_release.py`）

1. **Gap 2（核心，unified 模式的 enabler）**：`check_version_sync`（~1234）與 bump/「確認版本號」步驟用 `detect_version_files(root)`（只掃 root），改為優先採 `resolve_version_source(root, config)`（~334，已正確 honor `version_source.primary` 含子目錄）。使 monorepo 子目錄版本檔（如 `app/pubspec.yaml`）能被偵測、報告、bump。
2. **Gap 1**：`check_version_sync` 無條件呼叫 `check_version_sync_dual` 印「Chrome Extension 雙版本來源」。改為僅 `project_type == chrome-ext` 時印該 dual report；其餘印與 project_type 對應的版本源摘要。
3. **Gap 3**：`check_version_sync` 的分支慣例警告（~1281「慣例為 feature/v{mm}」）在 `config.release_workflow == "trunk"` 時跳過。
4. **Gap 5**：`check_worklog_completed`（~544）主日誌偵測：`v{version}-main.md` 不存在時，fallback 取版本子目錄內任一 `v{version}*.md` 為主日誌（而非直接 FAIL）。
5. **unified-monorepo 表達**：確保 `project_type: monorepo` + 頂層 `version_source`、無 `subprojects` 時，版本偵測/bump/preflight 全程以 version_source 為 SoT 走通（多數由改動 1 達成；補必要的 monorepo 分支判斷與文件 docstring）。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

於 tarrragon/claude clone（`/tmp/tcr-pr`，分支 `feat/version-release-unified-monorepo`）修改 `skills/version-release/scripts/version_release.py`，五項改動：

1. **Gap 2（核心 enabler）**：新增 helper `resolve_sync_version_files(root, config)`，優先採 `resolve_version_source`（honor `version_source.primary`，含子目錄如 `app/pubspec.yaml`），fallback 至 `detect_version_files(root)`。三處改用此 helper：`check_version_sync`、`verify_version_files`（「確認版本號」步驟）、`detect_version`（版本偵測/bump 來源）。
2. **Gap 1**：`check_version_sync` 改為僅 `project_type == chrome-ext` 時呼叫 `check_version_sync_dual` + `print_version_sync_report`；其餘類型直接走 version_source 摘要，不印誤導性 Chrome Extension 雙版本報告。
3. **Gap 3**：`check_version_sync` 的 feature 分支慣例警告於 `release_workflow == "trunk"` 時整段跳過（含 git branch 查詢）。
4. **Gap 5**：`check_worklog_completed` 主日誌偵測新增 fallback 1——`v{version}-main.md` 不存在時取版本子目錄內任一 `v{version}*.md` 為主日誌（原 fallback 2 舊結構保留）。
5. **unified-monorepo 表達**：`project_type: monorepo` + 頂層 `version_source`、無 `subprojects` 時，版本偵測/驗證/報告全程以 version_source 為 SoT 走通（由改動 1 + helper docstring 達成）。既有 `subprojects` 配置不受影響（向後相容）。

Gap 4（worklog 路徑跨 skill 不一致）未動，留 #4 討論。
