**所有改動與 git 操作都在 `/tmp/tcr-pr`，不要動 app_tunnel 本地。** 完成後 `git -C /tmp/tcr-pr push -u origin feat/version-release-unified-monorepo` 並 `gh pr create --repo tarrragon/claude` 引用 issue #4。

### 設計：unified-monorepo 模式

`project_type: monorepo` + 頂層 `version_source`（無 `subprojects`）= **統一版本模式**：version_source 指定的子專案版本檔為產品 SoT，tag `v{version}` 標整個 repo、bump 該檔。與既有 `subprojects`（獨立版本）並存、**向後相容**（既有 monorepo 配置用 subprojects，不受影響）。

### 五項改動（檔案：`skills/version-release/scripts/version_release.py`）

1. **Gap 2（核心，unified 模式的 enabler）**：`check_version_sync`（~1234）與 bump/「確認版本號」步驟用 `detect_version_files(root)`（只掃 root），改為優先採 `resolve_version_source(root, config)`（~334，已正確 honor `version_source.primary` 含子目錄）。使 monorepo 子目錄版本檔（如 `app/pubspec.yaml`）能被偵測、報告、bump。
2. **Gap 1**：`check_version_sync` 無條件呼叫 `check_version_sync_dual` 印「Chrome Extension 雙版本來源」。改為僅 `project_type == chrome-ext` 時印該 dual report；其餘印與 project_type 對應的版本源摘要。
3. **Gap 3**：`check_version_sync` 的分支慣例警告（~1281「慣例為 feature/v{mm}」）在 `config.release_workflow == "trunk"` 時跳過。
4. **Gap 5**：`check_worklog_completed`（~544）主日誌偵測：`v{version}-main.md` 不存在時，fallback 取版本子目錄內任一 `v{version}*.md` 為主日誌（而非直接 FAIL）。
5. **unified-monorepo 表達**：確保 `project_type: monorepo` + 頂層 `version_source`、無 `subprojects` 時，版本偵測/bump/preflight 全程以 version_source 為 SoT 走通（多數由改動 1 達成；補必要的 monorepo 分支判斷與文件 docstring）。

### 測試要求

- 既有測試不可破壞：`test_detect_project_type.py`、`test_new_functions.py`、`test_w1_127_config.py`、`test_w1_004_release_finalization.py` 全綠。
- 新增測試覆蓋：unified-monorepo（monorepo + version_source 無 subprojects → 以 version_source 偵測版本）、Gap 1（非 chrome-ext 不印 dual report）、Gap 3（trunk 跳過分支警告）、Gap 5（主日誌 fallback 命名）。
- 跑測試：`uv run --project skills/version-release pytest skills/version-release/tests/ -v`（或依該 skill pyproject 慣例）；N>=1 跑綠。
